Something Useful

Jan 13th, 2011

Well.

I worked on that scraper for a bit tonight, and finally have something that I will actually use now.  You can check it out HERE.  A few notes about it:

  • I did not go and learn the correct way to program in Ruby, as I claimed I would before finishing this. Although I did learn some basic Ruby syntax in the work I’ve done on this so far, I mostly grew impatient and decided to hack out a solution tonight.
  • No really, this is a hack. I decided I wanted sorting so I quickly found a javascript table sorter and based my output around that code (which is most noticeable with the * before January postings [yay alphabetic sorting!]).
    • The date was particularly problematic, considering that the page I was parsing gave dates in a %a %b %d format (i.e. ‘Thu Jan 10’). This meant I could either write my own code to figure out the year (which would still be a hack, and vary depending on the current date) or just hack it like I did (albeit temporarily)
  • Relating to that last point, I was considering storing this data in a database. This would make sorting a simple matter of changing the query, but that kind of stuff is easy and wouldn’t really help me or the page in any significant way; that’s probably not going to happen.
  • The script as it stands is just writing an HTML file from its data. While I could have had it do this in-place, I’m actually storing the jobs individually as objects in an array, and then later writing them out. This is obviously wasteful but again, at this scale who cares (as long as I’m aware of it, right?).
  • I set up the script to run every half-hour during normal business hours. I thought that was cool.

Scraping the Day Away

Jan 11th, 2011

So I’ve been tinkering with Ruby/Rails, using Mechanize to try and whip up a quick scraper for craigslist to aid me in my job search.

I know it’s not a novel or particularly interesting idea, but practice is practice, right?  I feel like I’m learning something and I’m definitely not comfortable with the language, so that’s good, right?

I mentioned earlier how I was impressed with how easily I was able to retrieve an array of link objects from a given page (seriously, 2 lines).  That still stands.  It seems that craigslist is particularly difficult for this kind of task, or at least this particular tool (in how I’ve come to know it in these past few hours).  The problem is that craigslist’s DOM ‘tree’ has lots of leaves.  These leaves are the data that I want.  So instead of using Mechanize’s neat features, I’m stuck basically just iterating through these leaves finding what I want.

That probably doesn’t make sense.  The point is, I need to learn more about all this before I actually try to write something.  In two hours I managed to write what’s basically just a compilation of several craigslist pages, and it works, but I know that this was a very, very naive approach.  I wrote a similar tool in Java over the summer for a job interview, and as it stands I’m not really using anything novel in the Ruby version (yet!).

The page is up on my site, but I’m not going to provide a link right now.  If you really cared (which you shouldn’t) you could probably find it.

This blog seems to have taken a somewhat negative turn as of the past few posts, so I wanted to say something about that:

The point of this blog is to chronicle my technical projects, my personal life, and my job search.  What may be thought of as negative is generally the more interesting topic, for me at least.  I was discussing the shortcomings of this Ruby project not because I dislike Ruby but because I find it to be more helpful/thought-provoking than writing about some small success in said project.

I think I’m done for the night.  Tomorrow’s goal: learn Ruby!

Oh Right. 2011.

Jan 8th, 2011

Second Saturday of the new year and I’m feeling just as lazy as 2010.  The difference?  Uhh … I’m writing this post!

Haven’t written in a couple weeks.  I went home for the holidays, which was nice.  Got to see lots of friends and family that I’ve not in a long time.  I also got a bike stand for easier home repairs, and most excitingly a Kindle.  I’ve had a lot of technical/educational documents on my computer for a while now, mostly in pdf, and I’m excited to finally have more opportunities to read them.

Made a little progress on Euler since we last talked, but I certainly haven’t hit my previous ‘one a day’ goal.  Did a few easy ones (16, 20) before I even went home for the holidays, and not much since.

It’s definitely fun to write these in an unfamiliar language (Python) and I might try a few other languages as well.

I’ve had a few ideas for little projects I want to work on.  One’s writing my own WP theme.  I think it’d be good practice, and would definitely help the feel of the blog vs. the main site, and I could add/remove functionality and links depending on what I am/am not using.  I mentioned before also wanting bypass any blog software completely and just write my own, but both of those are things I’m not in any rush to do.

The other and more pertinent idea is writing a craigslist scraper to do my daily craigslist perusals for me.  It certainly won’t be hard or really something I haven’t done before, so I was thinking about writing it in another language (I was thinking Ruby).

Thinking about that a little further, though, I failed to come up with any solid reason to do so, and found the same was true for any short-term project in a new language, for someone in my position.

My original motivation behind these little projects was threefold:  in writing them, I would be getting:

  • practice
  • with a new concept (say, web scraping)
  • in a new language (say, Ruby)

Then I asked myself: would any employer really care that I took a day to write a web scraper in Ruby? Sure, it shows motivation and enthusiasm, but couldn’t one say that about a college degree as well (at least, one in a hard science)?

Another point I was trying to prove, my ability to learn new languages, upon closer inspection also seems something that should be self-evident.  My college career was spent solving problems in languages A, B, and C.  My student job had me solving problems in languages D and E.  Should I really have to prove that I can solve problems in your language X, before being hired to do so?

I feel like that last point is the most frustrating part of my whole job search, and a serious shortcoming in regards to hiring practices for programmers.  The problem comes from these listings for ‘Junior/Entry-Level’ positions requiring anywhere from 2 to 7 years’ experience with a language.

Now, I know that I do not have 7 years’ experience with any language.   I feel uncomfortable saying I have 2 years’ experience with the languages I used in college (over 4 years’ time).  In reality, I have no professional experience with any language.

My problem has become: If I need this experience for an entry-level position … where do I enter?

Getting back to the original discussion, one day-long project in a new language is not going to qualify me for any job that I wasn’t already qualified for.

In fact, I don’t believe that any amount of programming would make me more qualified for a job than I am now.  That’s not to say that it wouldn’t make me a more productive worker at the beginning of hire, during what would be that training/learning period.  But how long would that period really be if you’re hiring the intelligent, competent worker you expect?

I guess it all comes down to those concepts of intelligence and competence.  Employers need indications of such, and look for it mainly in documented experience.  My problem lies with the fact that the college degree is no longer seen as such, for reasons I won’t get into here.

Don’t get me wrong, I’m still doing these projects.  I just no longer believe they’ll help me get a job.

Anyone have any ideas to help me get my foot in the door?
