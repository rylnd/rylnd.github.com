Euler, Oilier.

Jan 20th, 2011

What’s in the news …?

Off the top of my mind, this: LINK.

Now, I thought I was a patient person; I let programs run, pages load: I give them their time. But to surrender control of both my mouse and keyboard for two minutes made me reconsider just how impatient I really am at my desk.

I’m writing this as my solution to problem 12 on Project Euler chugs away, eating my resources. It’s a very, VERY brute-force solution, mind you. The actual question can be found here. It seems fairly straightforward; find the first triangle number with >500 factors. I wrote a method to calculate the nth triangle number, and then wrote a brute-force ‘factor counter’, which just goes through every number smaller than the given input and sees if it divides it cleanly. So that’s running.

In the meantime, though, I started looking up smarter solutions, of which this time there thankfully are. There’s a much easier way to calculate the number of factors a number has, based off its prime factors (I’ll leave you to google it if you’re interested). So, once I write a ‘prime-factorizer’ I should have a much better solution.

This work with Ruby and Python has given me more than an academic reason to learn more about functional programming. I watched a few really great FP talks by Rich Hickey that got me interested, but learning the basics was as far as that had got me. While there’s still not a pressing need to really involve myself, it’s nice to be able to slowly add bits and pieces to my repertoire as I need/research/find them.

EDIT(10:52pm) So … while I was writing that post, I was realizing that my brute-force solution wasn’t cutting it. In fact, it probably wouldn’t find the solution if I left it overnight, as it was taking exponentially longer with each new number (I was aware of this problem).

So, I went back, did a little research, and wrote a program that counts factors based on the number’s prime factors. It found the solution in under 1s. Problem 13 down.

Live and learn. Goodnight.

Something Useful

Jan 13th, 2011

Well.

I worked on that scraper for a bit tonight, and finally have something that I will actually use now.  You can check it out HERE.  A few notes about it:

  • I did not go and learn the correct way to program in Ruby, as I claimed I would before finishing this. Although I did learn some basic Ruby syntax in the work I’ve done on this so far, I mostly grew impatient and decided to hack out a solution tonight.
  • No really, this is a hack. I decided I wanted sorting so I quickly found a javascript table sorter and based my output around that code (which is most noticeable with the * before January postings [yay alphabetic sorting!]).
    • The date was particularly problematic, considering that the page I was parsing gave dates in a %a %b %d format (i.e. ‘Thu Jan 10’). This meant I could either write my own code to figure out the year (which would still be a hack, and vary depending on the current date) or just hack it like I did (albeit temporarily)
  • Relating to that last point, I was considering storing this data in a database. This would make sorting a simple matter of changing the query, but that kind of stuff is easy and wouldn’t really help me or the page in any significant way; that’s probably not going to happen.
  • The script as it stands is just writing an HTML file from its data. While I could have had it do this in-place, I’m actually storing the jobs individually as objects in an array, and then later writing them out. This is obviously wasteful but again, at this scale who cares (as long as I’m aware of it, right?).
  • I set up the script to run every half-hour during normal business hours. I thought that was cool.

Scraping the Day Away

Jan 11th, 2011

So I’ve been tinkering with Ruby/Rails, using Mechanize to try and whip up a quick scraper for craigslist to aid me in my job search.

I know it’s not a novel or particularly interesting idea, but practice is practice, right?  I feel like I’m learning something and I’m definitely not comfortable with the language, so that’s good, right?

I mentioned earlier how I was impressed with how easily I was able to retrieve an array of link objects from a given page (seriously, 2 lines).  That still stands.  It seems that craigslist is particularly difficult for this kind of task, or at least this particular tool (in how I’ve come to know it in these past few hours).  The problem is that craigslist’s DOM ‘tree’ has lots of leaves.  These leaves are the data that I want.  So instead of using Mechanize’s neat features, I’m stuck basically just iterating through these leaves finding what I want.

That probably doesn’t make sense.  The point is, I need to learn more about all this before I actually try to write something.  In two hours I managed to write what’s basically just a compilation of several craigslist pages, and it works, but I know that this was a very, very naive approach.  I wrote a similar tool in Java over the summer for a job interview, and as it stands I’m not really using anything novel in the Ruby version (yet!).

The page is up on my site, but I’m not going to provide a link right now.  If you really cared (which you shouldn’t) you could probably find it.

This blog seems to have taken a somewhat negative turn as of the past few posts, so I wanted to say something about that:

The point of this blog is to chronicle my technical projects, my personal life, and my job search.  What may be thought of as negative is generally the more interesting topic, for me at least.  I was discussing the shortcomings of this Ruby project not because I dislike Ruby but because I find it to be more helpful/thought-provoking than writing about some small success in said project.

I think I’m done for the night.  Tomorrow’s goal: learn Ruby!
